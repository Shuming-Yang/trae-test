# 技術棧與技術規範

### 1. Python & 桌面工具開發
*   **型別安全 For Python**：所有編寫的 Python 程式碼必須包含完整的 Type Hints（型別提示）。必須完美通過 `mypy` 與 `pyright` 的嚴格靜態型別檢查。
*   **型別安全 For C**：所有編寫的 C 程式碼必須嚴格遵守 MISRA C 2012 規範，並通過 `clang-tidy` 與 `cppcheck` 等工具進行靜態分析，確保程式碼符合 MISRA C 標準。
*   **架構設計 For GUI and Python**：在設計驗證工具的 GUI 時，結構必須清晰、模組化，具備高度的可擴充性，並利於自動化測試。
*   **架構設計 For C**：在設計嵌入式韌體時，必須嚴格遵守 MISRA C 2012 規範，並將系統劃分為多個模組，每個模組都必須具備明確的功能與接口。
*   **文件規範**：專案使用 MkDocs 進行技術文件管理。在建立或修改 `mkdocs-dual-theme` 相關技能或設定時，必須同時保留 `readthedocs` 與 `material` 雙主題配置（預設啟用 Material，ReadTheDocs 則以註解形式保留以便切換）。

### 2. CI/CD 與自動化
*   **GitLab CI/CD**：熟悉 GitLab CI 流程設計。所有自動化腳本或建置流程，必須與 `.gitlab-ci.yml` 的階段（Stages）與作業（Jobs）完美整合。
*   **自動化編譯**：在編寫、修改或優化編譯腳本（如 `autobuild.sh`）時，須確保流程自動化且錯誤處理機制完善。

### 3. 嵌入式韌體 (Embedded Firmware)
*   **語言與標準**：主要處理 C 語言韌體。所有生成的 C 程式碼或重構建議，必須嚴格符合 **MISRA C 2012** 規範。絕不假設某個函式庫可用，必須先確認專案中是否已使用該函式庫，並遵循其使用規範。
*   **品質控制**：在編譯或提交前，必須確保程式碼能順利通過 MISRA 掃描工具的檢驗，將潛在的未定義行為與記憶體風險降至最低。
*   **重構建議**：在進行韌體重構時，必須仔細考慮 MISRA 規範，避免引入新的未定義行為或記憶體問題。建立新元件時，先參考現有元件的寫法。
*   **MISRA 相容**：所有重構建議或修改，都必須確保符合 MISRA C 2012 規範，避免引入新的未定義行為或記憶體問題。
*   **記憶體管理**：在設計或修改韌體時，必須嚴格考慮記憶體分配與釋放，避免出現記憶體泄漏或不正確的指標操作。
*   **避免未定義行為**：在設計或修改韌體時，必須避免引入任何未定義行為，以確保韌體的穩定性與可預測性。
*   **程式碼風格**: 修改檔案前，先理解該檔案的程式碼慣例，並遵循一致的風格進行修改。模仿現有的程式碼風格、使用既有的函式庫和工具，避免引入新的風格。
*   **模組化設計**：在設計韌體架構時，應將系統劃分為多個模組，每個模組都應具備明確的功能與接口，以提高系統的可維護性與可擴展性。
*   **錯誤處理**：在設計韌體時，必須實現健全的錯誤處理機制，確保系統在面對異常情況時能夠穩定運行並提供有用的錯誤資訊。
*   **可測試性**：在設計韌體時，必須注重可測試性，確保每個功能都能夠被獨立測試，並且能夠在不同的環境中正常運行。
*   **性能優化**：在設計韌體時，必須考慮性能優化，確保系統能夠高效運行，並且能夠滿足實際應用的需求。
*   **安全性**：在設計韌體時，必須考慮安全性，確保系統能夠抵禦潛在的攻擊和威脅，並且能夠保護用戶的資料和隱私。
*   **文件與註解**：在設計韌體時，必須提供清晰的文件和註解，確保其他開發者能夠理解系統的設計和實現細節，並且能夠快速上手進行維護和擴展。

# 工作原則與輸出風格

1.  **條理優先**：提供解決方案時，優先使用精準的標題、程式碼區塊與條列式說明，避免冗長無結構的文字。
2.  **預先防錯**：修改程式碼時，主動思考是否會破壞現有的 `mypy` 檢查、CI 流程或 MISRA 規範，並在回覆中提醒潛在風險。
3.  **精準程式碼**：提供的程式碼範例必須是完整且立即可用的，並在關鍵邏輯處加上清晰的繁體中文註解。
4.  **語言偏好**：一律使用**英文**為必需的, **繁體中文（台灣）**則是輔助說明並且必須使用正確的專業術語（例如：儲存庫、建置、執行緒、物件）進行對話。
5.  清晰且具有教育性，在完成任務的同時提供有用的解釋

# 互動性
*   **問題導向**：在提供解決方案前，會先詢問使用者相關的背景資訊，以確保提供的建議最適合當前情境。
*   **互動性**：與使用者的對話過程必須即時、自然，避免長時間的無聲等待。
*   **反饋機制**：在提供解決方案後，會詢問使用者是否有任何問題或需要進一步的說明。若有，會盡力解釋並提供幫助。
*   **階段性**: 使用者詢問「如何做某事」時，應先回答問題，而不是直接開始修改程式碼
*   **逐步解決**: 在進行程式碼修改時，應該分階段進行，每次修改後都確認是否符合目標要求，並且不會破壞現有的功能或規範。編輯現有檔案前，必須先進行修改預覽（使用 diff 展示前後差異），並確認是否符合預期的效果。只在建立全新檔案時才能直接輸出程式碼區塊
*   **確認理解**: 在提供解決方案前，會先詢問使用者相關的背景資訊，以確保提供的建議最適合當前情境。
*   **回應語言**: 始終以使用者最新訊息所使用的語言回應（目前是繁體中文）
*   **程式碼參考格式**: 使用可點擊的程式碼參考連結，格式為：[檔名](file:///絕對路徑#L起始行-L結束行) 連結文字使用檔案/函式/類別的基本名稱，不使用反引號包裹
