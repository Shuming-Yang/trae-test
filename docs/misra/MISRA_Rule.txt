Appendix A Summary of guidelines
Dir 1.1 required
Dir1.1 [Required] Any implementation-defined behaviour on which the output of the program depends shall be documented and understood.
Directive 2.1 required
Dir2.1 [Required] All source files shall compile without any compilation errors.
Directive 3.1 required
Dir3.1 [Required] Allcode shall be traceable to documented requirements.
Directive 4.1 required
Dir4.1 [Required] Run-time failures shall be minimized.
Directive 4.2 advisory
Dir4.2 [Advisory] All usage of assembly language should be documented.
Directive 4.3 required
Dir4.3 [Required] Assembly language shall be encapsulated and isolated.
Directive 4.4 advisory
Dir4.4 [Advisory] Sections of code shoud not be "commented out"
Directive 4.5 advisory
Dir4.5 [Advisory] Identifiers in the same name space with overlapping visibility should be typographically unambiguous.
Directive 4.6 advisory
Dir4.6 [Advisory] typedefs that indicate size and signedness should be used in place of the basic numerical types.
Directive 4.7 required
Dir4.7 [Required] If a function returns error information, then that error information shall be tested.
Directive 4.8 advisory
Dir4.8 [Advisory] If a pointer to a structture or union is never dereferenced within a translation unit, then the implementation of the object should be hidden.
Directive 4.9 advisory
Dir4.9 [Advisory] A function shall be used in preference to a function-like macro where they are interchangeable.
Directive 4.10 required
Dir4.10 [Required] Precautions shall be taken in order to prevent the contents of a header file being included more than once.
Directive 4.11 required
Dir4.11 [Required] The validity of values passed to library functions shall be checked.
Directive 4.12 required
Dir4.12 [Required] Dynamic memory allocation shall not be used.
Directive 4.13 advisory
Dir4.13 [Advisory] Functions which are designed to provide operation on a resource should be called in an appropriate sequence.
Directive 4.14 required
Dir4.14 [Required] The validity of values received from external sources shall be checked

R2.7 [Advisory] There should be no unused parameters in functions

R3.1 [Required] The character sequences /* and // shall not be used within a comment.
Rule 3.2 required
R3.2 [Required] Line-splicing shal not be used in // comments
Rule 4.1 required
R4.1 [Required] Octal and hexadecimal escape sequences shall be terminated.
Rule 4.2 advisory
R4.2 [Advisory] Trigraphs should not be used.
Rule 5.1 required
R5.1 [Required] External identifiers shall be distinct.
Rule 5.2 required

R5.4 [Required] Macro identifiers shall be distinct
Rule 5.5 required
R5.5 [Required] Identifiers shall be distinct from macro names
Rule 5.6 required
R5.6 [Required] A typedef name shall be a unique identifier.
Rule 5.7 required
R5.7 [Required] A tag name should be a unique identifier.
Rule 5.8 required
R5.8 [Required] Identifier that define objects or function with external linkage shall be unique.
Rule 5.9 advisory
R5.9 [Advisory] Identifiers that define objects or functions with internall linkage should be unique.
Rule 6.1 required
R6.1 [Required] Bit-fields shall only be declared with an appropriate type.
Rule 6.2 required
R6.2 [Required] Single-bit named bit fields shall not be of a signed type.
Rule 7.1 required
R7.1 [Required] Octal constants shall not be used.
Rule 7.2 required
R7.2 [Required] A “u” or “U” suffix shall be applied to all integer constants that are represented in an unsigned type.
Rule 7.3 required
R7.3 [Required] The lowercase character 'l' shall not be used in a literal suffix.
Rule 7.4 required
R7.4 [Required] A string literal shall not be assigned to an object unless the object's type is "pointer to const-qualified char"
Rule 8.1 required
R8.1 [Required] Types shall be explicitly specified.
Rule 8.2 required
R8.2 [Required] Function types shall be in prototype form with named parameters
Rule 8.3 required
R8.3 [Required] All declarations of an object or function shall use the same names and type qualifiers.
Rule 8.4 required
R8.4 [Required] A compatible declaration shall be visible when an object or function with external linkage is defined.
Rule 8.5 required
R8.5 [Required] An external object or function shall be declared once in one and only one file.
Rule 8.6 required
R8.6 [Required] An identifier with external linkage shall have exactly one external definition.
Rule 8.7 advisory
R8.7 [Advisory] Functions and objects should not be defined with external linkage if there are referenced in only one translation unit.
Rule 8.8 required
R8.8 [Required] The static storage class specifier shall be used in al deciarations of objects and functions that have internal linkage.
Rule 8.9 advisory
R8.9 [Advisory] An object should be defined at block scope if its identifier only appears in a single function.
Rule 8.10 required
R8.10 [Required] An inline funciton shall be declared with the static storage class.
Rule 8.11 advisory
R8.11 When an array with external linkage is declared. Its size should be explicit specified.
Rule 8.12 required
R8.12 [Required] Within an enumeration list , the value of an implicitly-specified enumeration constant shall be unique.
Rule 8.13 advisory
R8.13 [Advisory] A pointer should point to a const-qualified type whenever possible
Rule 8.14 required
R8.14 [Required] The restrict type qualifier shall not be used.
Rule 9.1 mandatory
R9.1 [Mandatory] The value of an object with automatic storage duration shall not be read before it has beed set.
Rule 9.2 required
R9.2 [Required] The initializer for an aggregate or union shall be enclosed in braces.
Rule 9.3 required
R9.3 [Required] Arrays shall not be partially initialized.
Rule 9.4 required
R9.4 [Required] An element of an object shall not be initialized more than once.
Rule 9.5 required
R9.5 [Required] Where designated initializaers are used to initialize an array object the size of the array shall be specified explicitly.
Rule 10.1 required
R10.1 [Required] Operands shall not be of an inappropriate essential type. The value of an expression of integer type shall not be implicitly converted to a different underlying type if: (a) it is not a conversion to a wider integer type of the same signedness, or (b) signedness, or the expression is complex, or (c) the expression is not constant and is a function argument, or (d) the expression is not constant and is a return expression 
Rule 10.2 required
R10.2 [Required] Expressions of essentially character type shall not be used.
Rule 10.3 required
R10.3 [Required] The value of an expression shall not be assigned to an object with a narrower essential type or of a different essential type category.
Rule 10.4 required
R10.4 [Required] Both operands of an operator in which the usual arithmetic conversions are performed shall have the same essential type category.
Rule 10.5 advisory
R10.5 [Advisory] The value of an expression should not be cast to an inappropriate essential type.
Rule 10.6 required
R10.6 [Required] The value of a composite expression shall not be assigned to  an object with wider essential type.
Rule 10.7 required
R10.7 [Required] If a composite expression is used as one operand of an operator in which the usual arithmetic conversions are performed then the other operand shall not have wider essential type.
Rule 10.8 required
R10.8 [Required] The value of a composite expression shall not be cast to a different essential type category or a wider essential type.
Rule 11.1 required
R11.1 [Required] [IGNORE] conversion between a pointer to function and another type
Rule 11.2 required
R11.2 [Required] Conversions shall not be performed between a pointer to an incomplete type and any other type.
Rule 11.3 required
R11.3 [Required] A cast shall not be performed between a pointer to object type and a pointer to a different object type.
Rule 11.4 advisory
R11.4 [Advisory] A conversion should not be performed between a pointer to object and an integer type.
Rule 11.5, advisory
R11.5 [Advisory] A conversion should not be performed from pointer to void into pointer to object.
Rule 11.6 required
R11.6 [Required] [IGNORE] A cast shall not be performed between pointer to void and an arithmetic type.
Rule 11.7 required
R11.7 [Required] A case shall not be performed between pointer to object and a non-integer arithmetic type.
Rule 11.8 required
R11.8 [Required] A cast shall not remove any const or volatile qualification from the type pointed to by a pointer.
Rule 11.9 required
R11.9 [Required] The macro NULL shall be the only permitted form of integer null pointer constant.
Rule 12.1 advisory
R12.1 [Advisory] The precedence of operators within expression should be made explicit

R12.2 [Required] The right hand operand of a shift operator shall lie in the range zero to one less than the width in bits of the essential type of the left hand operand.

R13.1 [Required] Initialzer lists shall not contain persistent side effects.

Rule 13.4 advisory
R13.4 [Advisory] The result of an assignment operator should not be used.
